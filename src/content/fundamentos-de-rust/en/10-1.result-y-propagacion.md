# Result, unwrap, and the `?` operator

## Explicit handling with `match`

The most explicit way to handle a `Result`:

```rust
use std::num::ParseIntError;

fn parse(s: &str) -> Result<i32, ParseIntError> {
    s.trim().parse::<i32>()
}

fn main() {
    match parse("42") {
        Ok(n)  => println!("Number: {n}"),
        Err(e) => println!("Error: {e}"),
    }
}
```

## `unwrap` and `expect`

For rapid prototyping or when you're sure there's no error:

```rust
fn main() {
    // unwrap: panic if Err
    let n: i32 = "42".parse().unwrap();

    // expect: panic with custom message
    let m: i32 = "42".parse().expect("Expected an integer");

    println!("{n} {m}");
}
```

**When to use:**
- `unwrap` / `expect`: in tests, examples, and cases where the error is impossible.
- Never in critical production code — propagate the error instead.

## The `?` operator

`?` is syntactic sugar for error propagation. It's equivalent to:

```rust
let value = match result {
    Ok(v)  => v,
    Err(e) => return Err(e.into()),
};
```

Use it to write linear code without excessive nesting:

```rust
use std::fs::File;
use std::io::{self, Read};

fn read_file(path: &str) -> Result<String, io::Error> {
    let mut file = File::open(path)?;       // propagates if it fails
    let mut content = String::new();
    file.read_to_string(&mut content)?;     // propagates if it fails
    Ok(content)
}
```

Only works inside functions that return `Result` (or `Option`).

## Custom errors

Define your own error type with an enum:

```rust
use std::fmt;

#[derive(Debug)]
enum ValidationError {
    InvalidAge(i32),
    EmptyName,
    InvalidEmail(String),
}

impl fmt::Display for ValidationError {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        match self {
            ValidationError::InvalidAge(e) => write!(f, "invalid age: {e}"),
            ValidationError::EmptyName     => write!(f, "name cannot be empty"),
            ValidationError::InvalidEmail(m) => write!(f, "invalid email: {m}"),
        }
    }
}

fn validate_age(age: i32) -> Result<(), ValidationError> {
    if age < 0 || age > 150 {
        Err(ValidationError::InvalidAge(age))
    } else {
        Ok(())
    }
}

fn main() {
    match validate_age(200) {
        Ok(())  => println!("Valid age"),
        Err(e)  => println!("Error: {e}"),
    }
}
```

## Transforming errors with `map_err`

When you need to convert the error type:

```rust
fn parse_age(s: &str) -> Result<u32, String> {
    s.trim()
     .parse::<u32>()
     .map_err(|_| format!("'{s}' is not a valid age"))
}
```

## `panic!` — unrecoverable errors

Use `panic!` only when the program has reached an unrecoverable state:

```rust
fn divide(a: i32, b: i32) -> i32 {
    if b == 0 {
        panic!("Division by zero — this should never happen");
    }
    a / b
}
```

> **General rule:** use `Result` for errors the caller can handle. Use `panic!` for programmer bugs that should never occur in correct code.
