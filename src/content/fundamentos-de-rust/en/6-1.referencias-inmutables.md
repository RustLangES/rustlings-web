# Immutable References

A reference (`&T`) allows accessing a value without taking its ownership. The `&` creates a reference; this process is called *borrowing*.

## Basic borrowing

```rust
fn calculate_length(s: &String) -> usize {
    s.len()
} // s is a reference, not the owner — nothing is freed

fn main() {
    let s1 = String::from("hello");

    let len = calculate_length(&s1); // we borrow s1

    println!("The length of '{s1}' is {len}"); // s1 is still valid
}
```

## Multiple immutable references

You can have **as many immutable references as you want** at the same time:

```rust
fn main() {
    let s = String::from("hello");

    let r1 = &s;
    let r2 = &s;
    let r3 = &s;

    println!("{r1} {r2} {r3}"); // all valid simultaneously
}
```

## References don't modify the value

Immutable references are read-only:

```rust
fn try_to_modify(s: &String) {
    // s.push_str("!"); // ERROR: cannot borrow `*s` as mutable
    println!("{s}"); // read-only: OK
}
```

## Why use references instead of moving

```rust
// Without references: need to return the String to keep using it
fn length_without_ref(s: String) -> (String, usize) {
    let len = s.len();
    (s, len) // we return s so we don't lose it
}

// With references: much cleaner
fn length(s: &String) -> usize {
    s.len()
}

fn main() {
    let s = String::from("hello");
    let len = length(&s); // s is still ours
    println!("'{s}' has {len} characters");
}
```

## `&str` vs `&String`

For function parameters, prefer `&str` over `&String` — it accepts both `String` and literals:

```rust
fn greet(name: &str) {
    println!("Hello, {name}!");
}

fn main() {
    let s = String::from("Alice");
    greet(&s);           // &String coerces to &str
    greet("Bob");        // &str directly
}
```

> **Rule:** use `&str` in function parameters when you only need to read text. It's more flexible than `&String`.
